## Key Architectural Decisions Locked In

### Monorepo topology (ADR-0026 Decision §)

- Bun workspace with `app/` (customer, was repo root) + `admin/` (operator) + `worker/` + `packages/*` + shared `supabase/`, `docs/`, `tests/`.
- Two Vercel projects, one per app folder, both linked to the same Supabase project + same Cloudflare Worker. Independent deploy cadences.
- Cloudflare Access in front of `admin.consentshield.in` (free tier, GitHub-OAuth-restricted to Sudhindra) as defence-in-depth on top of Supabase Auth + AAL2.
- Reversible until Phase 4 (Vercel project split is point of no return).

### Narrow package sharing (Part 6 of this session; in feedback memory)

- Only 3 shared packages: `shared-types`, `compliance`, `encryption`.
- NOT shared: Supabase clients (server + browser), shadcn/ui components, app-specific lib code. Each app keeps its own copy.
- Rationale: independence + smaller blast radius + hard security boundary between operator-privileged code and customer-reachable code.

### Admin auth (admin platform doc §3)

- Same `auth.users` table as customer; differentiator is `app_metadata.is_admin = true` JWT claim + `app_metadata.admin_role` (platform_operator | support | read_only).
- Hardware key required (Supabase WebAuthn / passkey). Admin app's `proxy.ts` rejects any session with `aal != 'aal2'`.
- 3 admin role tiers; impersonation requires reason ≥ 10 chars + customer notification within 5 min + 30-min default duration.

### Admin schema pattern (ADR-0027 Decision §)

- New `cs_admin` Postgres role with `BYPASSRLS = true` for SELECT only.
- New `admin` schema with 11 tables: admin_users, admin_audit_log (partitioned monthly, append-only), impersonation_sessions, sectoral_templates, connector_catalogue, tracker_signature_catalogue, support_tickets + messages, org_notes, feature_flags, kill_switches (4 default switches seeded), platform_metrics_daily.
- Every admin write to a customer table goes through a security-definer RPC in `admin.*` that audit-logs in same transaction (Rule 22).
- Impersonation grants RLS-context **read** only; writes during impersonation still go through audit-logged RPCs.

### Admin Rules 21–25 (added to admin platform doc §9)

- 21: Hardware key 2FA required for all admin sessions
- 22: Every admin action audit-logged in same transaction
- 23: Impersonation time-boxed + reason-required + customer-notified within 5 min
- 24: No admin endpoint reachable from customer subdomain (host check in admin/proxy.ts)
- 25: Admin app deploys independently from customer app (separate Vercel project, separate Sentry, separate env vars)

### Workflow rule: wireframes before ADRs for UI features (Part 5 of this session; in feedback memory)

- For any UI feature: wireframe → architecture → alignment → ADR → code.
- For pure-infrastructure ADRs (no UI): wireframe step is skipped; ADR comes first.
