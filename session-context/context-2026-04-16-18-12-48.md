## Key Architectural Decisions Locked In

### Q1 — `consent_artefacts` data category
**Option B accepted.** Category A (operational) with an orthogonal "delivered to customer storage" property. No new category. Other Category A tables (rights_requests, retention_rules, consent_artefact_index) carry the same orthogonal property.

### Q2 — Artefact-creation atomicity
**Option D (hybrid) accepted.** AFTER INSERT trigger on `consent_events` calls `net.http_post()` to `process-consent-event` Edge Function (sub-second typical latency, same primitive as existing HTTP cron jobs). pg_cron safety net every 5 minutes sweeps orphan events. Both paths share an idempotent Edge Function (skip creation if artefacts exist for `consent_event_id`). Trigger body wrapped in `EXCEPTION WHEN OTHERS THEN NULL`.

### Rule 3 broadened
From "Health data (FHIR) is never persisted" to "Regulated sensitive content is never persisted" — covers FHIR + banking identifiers (PAN, Aadhaar, account numbers, transaction records) + future sectors. `data_scope` is a category declaration (`'pan'`, `'email_address'`), never an actual value. Category C renamed.

### Rules 19 + 20 added
- **Rule 19:** consent_artefacts append-only — status transitions only via `artefact_revocations` INSERT trigger, `enforce_artefact_expiry` pg_cron, or process-consent-event re-consent path.
- **Rule 20:** every consent artefact has non-null `expires_at` populated from `purpose_definitions.default_expiry_days`. Open-ended consent prohibited.

### S-5 replacement chain semantics
If artefact A is replaced by B and B is later revoked, A stays frozen at `replaced`. Revocation does NOT walk `replaced_by`. The chain is a historical record of how consent was re-obtained, not a live authorisation chain.

### Phase B Point 1 (no legacy in data model)
Every banner purpose MUST carry `purpose_definition_id`. Banner save/publish endpoints return 422 if missing. `process-consent-event` writes `consent_events_misconfigured` audit + P1 alert + zero artefacts. `coverage_score` expected 100% at all times. NOT a statement about schema objects — those evolve via ALTER in §11.3.

### Phase C migration note
ALTER-in-place chosen for all 6 §11 amendments to existing objects (rationale documented per amendment in §11.13). DROP+RECREATE available as fallback for future amendments where ALTER would be awkward.
