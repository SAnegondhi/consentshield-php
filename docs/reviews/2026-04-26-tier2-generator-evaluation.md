# Tier-2 SDK generator evaluation — OpenAPI Generator vs Speakeasy vs Stainless

**Date:** 2026-04-26
**ADR:** ADR-1028 (Phase 1 Sprint 1.1)
**Decision:** Stay on OpenAPI Generator for v1.0.0; revisit at $50k MRR or first BFSI Tier-1 procurement comment on code quality.

---

## Scope

Compare generator output for the same input — `app/public/openapi.yaml` (21 operations, OpenAPI 3.1.0, Bearer-only auth) — across three candidate tools, for three target languages (Java, C#, PHP).

| Tool | Cost | Spec ownership | Output cleanliness (claimed) | Re-evaluation gate |
|---|---|---|---|---|
| OpenAPI Generator (`openapitools/openapi-generator-cli:v7.10.0`) | Free, OSS | Self-hosted | Industrial — same shape `swagger-codegen` has emitted for a decade | n/a (default) |
| Speakeasy | Free OSS tier (qualifies — public spec); paid for private + advanced features | Spec uploaded to Speakeasy build | Hand-tuned templates, idiomatic per language | $50k MRR OR first BFSI Tier-1 RFP commenting on code quality |
| Stainless | Paid only ($2k+/mo) | Spec uploaded to Stainless build | Stripe-quality templates | Out of scope until Speakeasy proves insufficient |

---

## Why OpenAPI Generator was chosen for v1.0.0

1. **Zero ongoing cost.** Pre-revenue, a free-OSS generator removes a recurring SaaS bill from the runway.
2. **Spec stays in-repo.** The OpenAPI source-of-truth at `app/public/openapi.yaml` is regenerated locally + in CI. No third-party service holds the spec or gates the build pipeline.
3. **The output is "good enough" for Tier-2 procurement.** AWS, Twilio, and several other Tier-1 vendors ship Java/.NET/PHP via OpenAPI Generator (or close variants). Procurement teams accept this shape because they've reviewed it before. A hand-tuned Speakeasy/Stainless client is *prettier* but does not clear procurement gates that OpenAPI-Generator output cannot.
4. **Reversible.** Input is the same OpenAPI spec; swapping the generator is a one-CI-step change. The decision can be reversed without re-architecting anything customer-facing.
5. **Compliance contract is hand-written either way.** Tier-2 SDKs need a hand-written wrapper layer (Spring Boot auto-config / ASP.NET DI / PSR-18) to enforce fail-CLOSED + retry-on-5xx-only + 4xx-always-throws. That code is identical regardless of the generator. Generator polish only affects the *body* of the generated API methods, not the compliance-load-bearing wrappers.

## What was actually compared

### OpenAPI Generator output

Generated by `bun run generate:tier2-sdks` (= `bunx tsx scripts/generate-tier2-sdks.ts`).

- Java: `packages/java-client/generated/` — Maven layout, `okhttp-gson` library.
- .NET: `packages/dotnet-client/generated/` — `httpclient` library, .NET 8.0 LTS.
- PHP: `packages/php-client/generated/` — Composer layout, default Guzzle wrapper.

Sample artefacts inspected for each target:
- Method-count vs operation-count parity.
- Bearer-auth handling (must read `cs_live_*` from configuration, not hard-coded).
- Error model (RFC 7807 problem+json on 4xx — must round-trip `type/title/status/detail/instance`).
- `X-CS-Trace-Id` header round-trip — generator produces request-side; response-side echo is documented as a separate ADR-1014 follow-up (not blocked here).

## Decision

**Stay on OpenAPI Generator for v1.0.0** across all three Tier-2 targets. Re-evaluation gate:

1. **MRR threshold:** $50k MRR sustained for 3 consecutive months. Indicates customer base capable of expressing preferences via revenue.
2. **Procurement signal:** any BFSI Tier-1 enterprise prospect explicitly comments on generated-client code quality during a real RFP (not "would be nicer if" feedback — actual procurement-blocking comment).
3. **Maintenance pain:** if the OpenAPI Generator output starts breaking compliance-contract tests in non-trivial ways across multiple sprints — i.e. the generator output is fighting the wrapper layer rather than complementing it.

If any of those triggers fires, reopen this review and switch generators. The switch is a one-CI-step change; the input is the same OpenAPI spec; the hand-written wrappers (Spring Boot / ASP.NET DI / PSR-18) survive the swap.
