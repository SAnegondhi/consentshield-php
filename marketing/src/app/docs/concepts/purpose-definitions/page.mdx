# Purpose definitions

A **purpose** is the reason the Data Fiduciary processes a specific piece
of personal data. DPDP §6(1) requires the Fiduciary to publish this
before collection — in the notice the Data Principal reads, in plain
language, including retention, legal basis, and any downstream transfers.

Every consent artefact points back to exactly one purpose row. The
ConsentShield API treats purposes as **versioned, immutable records**:
when a purpose changes materially, a new version is published, and
in-flight artefacts continue to reference the version the principal
actually consented to.

## Reading purposes from the API

Every integration starts with listing the purposes that apply to its
organisation. The list comes from the sectoral template you applied at
onboarding plus any custom purposes you've added.

## Versioning

Purposes are **immutable once published**. If you change the description
or expand the `data_scope`, the system publishes a new version (e.g.
v3 → v4). Existing artefacts continue to point at v3 — the legal basis
under which they were collected hasn't changed.

<Callout tone="info" title="When to publish a new version">
**Do** publish a new version when the description changes materially,
when <code>data_scope</code> grows, when <code>recipients</code> change,
or when <code>retention_days</code> shortens. **Don't** publish for cosmetic
changes (typo fixes, translation updates) — ConsentShield treats those
as version metadata, not a material change.
</Callout>

When a new version is material, existing artefacts can stay under the
old version until their next natural touch-point (next login, next
rights-request, etc.), at which point the principal sees the updated
notice and re-consents.

<Callout tone="warn" title="Material-change re-consent">
For banners, ConsentShield re-prompts when <code>purpose.version</code>
increments. For in-app preference centers, you're responsible for
triggering the re-consent flow — the API returns
<code>verify_status = &quot;superseded&quot;</code> when an artefact's
version is older than the current purpose version. Treat that as
<code>needs_reconsent</code> in your UI.
</Callout>

## Sectoral templates

ConsentShield ships **sectoral templates** — pre-composed purpose
bundles for common sectors. BFSI, Healthcare/ABDM, SaaS, E-commerce.
At onboarding you pick a template; the purposes land pre-published in
your organisation.

You can edit the defaults, add custom purposes, or swap to a different
sectoral template from **Settings → Sector template**. The account-
default sectoral template (ADR-1027 Sprint 3.3) pre-selects the
operator's preferred template for every new org in an enterprise
account — useful for groups deploying across multiple divisions.
