# Artefacts vs. events

The ConsentShield API distinguishes two kinds of record. Understanding
which one to call — and when the system produces them on your behalf — is
the single biggest lift for a new integrator.

## TL;DR

<ParamTable params={[
  {
    name: 'Consent artefact',
    type: 'durable',
    description: 'A first-class, immutable, DPB-quotable record of a specific consent decision. Has a stable id. Lives in the customer\'s R2/S3 bucket. Queried via /v1/consent/verify, /v1/consent/artefacts.',
  },
  {
    name: 'Consent event',
    type: 'transient',
    description: 'A single state-transition row: recorded, revoked, expired, replaced. Buffered for seconds to minutes before delivery to customer storage; deleted from ConsentShield afterwards. Queried via /v1/consent/events for recent history.',
  },
]} />

A single artefact typically has **N events** attached: one `recorded`, zero
or more `replaced` (when a new version supersedes it), and optionally a
`revoked` or `expired`. The artefact is the **subject**; the events are the
**verbs** that happened to it.

## Why this split exists — the stateless oracle

ConsentShield is architecturally a **stateless compliance oracle**:

1. A consent decision arrives at the API (`POST /v1/consent/record`).
2. An artefact is materialised in-memory, signed, and written to the
   customer's own R2 or S3 bucket.
3. A corresponding event row is written to ConsentShield's buffer.
4. A background pipeline delivers the buffered rows to customer
   storage within minutes.
5. **Once delivery is confirmed, the ConsentShield row is deleted.**

The customer ends up holding the canonical record. ConsentShield holds
only what's needed to serve the next verify / revoke / list request —
and even that is bounded.

<Callout tone="info" title="Why does this matter to my integration?">
It means you can't rely on ConsentShield as your long-term data store.
Audit exports, DPB responses, and historical reports all read from your
R2/S3 bucket directly — we just help you produce and organise the
artefacts. [Build a DPB audit export](/docs/cookbook/build-dpb-audit-export)
walks through this path.
</Callout>

## The artefact life cycle

```
 POST /v1/consent/record
          │
          ▼
 ┌─────────────────────┐
 │  artefact created   │   status = active
 │  (delivered to R2)  │   events  = [recorded]
 └─────────┬───────────┘
           │
           │  POST /v1/consent/record   — user consents to a superseding version
           │  ───────────────────────▶  events += [replaced]
           │
           │  POST /v1/consent/artefacts/{id}/revoke
           │  ───────────────────────▶  status = revoked · events += [revoked]
           │
           │  purpose.auto_delete_on_expiry + time
           │  ───────────────────────▶  status = expired  · events += [expired]
           │
           ▼
 ┌─────────────────────┐
 │   terminal state    │   Still queryable via /consent/verify?artefact_id=…
 │                     │   and /consent/artefacts for audit.
 └─────────────────────┘
```

## What lives in the buffer, what lives in R2

<Callout tone="warn" title="The buffer is not your archive">
Do **not** build reports that read from ConsentShield's `/v1/consent/events`
beyond a 90-day window. The buffer is a write-ahead log; we delete
delivered rows within minutes. Historical queries of any age should
read from your own R2/S3 bucket, where every event and every artefact
has been durably delivered.
</Callout>
