# Quickstart — record your first consent artefact

In 15 minutes you will issue an API key, call `POST /v1/consent/record`, and
verify the resulting artefact. If you get stuck, email
[dev-support@consentshield.in](mailto:dev-support@consentshield.in) — we
reply within one business day.

<Callout tone="info" title="Before you start">
You'll need a ConsentShield account ([sign up here](https://consentshield.in/signup)).
The trial includes a sandboxed organisation where every request below will
succeed without a payment method.
</Callout>

## 1. Issue an API key

In the dashboard go to **Settings → API keys → New key**. Give it a name
(e.g. *staging-integration*) and copy the plaintext. **You will only see
it once** — store it in a secret manager, never in source control.

The key looks like `cs_live_01HRG…N7QZ`. Keys are scoped to one
organisation; a separate `org_id` is embedded inside and verified on every
request.

<Callout tone="security" title="Never commit keys">
`cs_live_*` keys grant full tenant API access. A leaked key should be
rotated immediately via **Settings → API keys → Rotate** — see
[Rotate an API key safely](/docs/cookbook/rotate-api-key-safely) for the
zero-downtime procedure.
</Callout>

Call the health endpoint to confirm authentication works.

console.log(res.status, body) // 200 { ok: true }`,

r = httpx.get(

A non-200 response means the key is wrong, revoked, or missing the
`Bearer ` prefix — see [authentication](/docs/authentication).

## 3. Record a consent

The minimum viable request needs a data-principal identifier, a purpose
code (fetched via `GET /v1/purposes`), and the consent outcome. Every
`POST /v1/consent/record` is **idempotent** when you supply an
`Idempotency-Key` header — the same key replayed returns the same artefact
id.

<CodeTabs tabs={[
  {
    label: 'cURL',
    language: 'bash',
    code: `curl -X POST "https://api.consentshield.in/v1/consent/record" \\
  -H "Authorization: Bearer $CS_API_KEY" \\
  -H "Content-Type: application/json" \\
  -H "Idempotency-Key: order-78421-marketing" \\
  -d '{
    "data_principal": { "email": "a@example.in" },
    "purpose_code":   "marketing_email",
    "outcome":        "granted",
    "origin":         "https://shop.example.in/checkout"
  }'`,
  },
  {
    label: 'Node.js',
    language: 'typescript',
    code: `const res = await fetch(
  'https://api.consentshield.in/v1/consent/record',
  {
    method: 'POST',
    headers: {
      Authorization: \`Bearer \${process.env.CS_API_KEY}\`,
      'Content-Type': 'application/json',
      'Idempotency-Key': 'order-78421-marketing',
    },
    body: JSON.stringify({
      data_principal: { email: 'a@example.in' },
      purpose_code: 'marketing_email',
      outcome: 'granted',
      origin: 'https://shop.example.in/checkout',
    }),
  },
)
const { artefact_id } = await res.json()`,
  },
  {
    label: 'Python',
    language: 'python',
    code: `import os, httpx

r = httpx.post(
    "https://api.consentshield.in/v1/consent/record",
    headers={
        "Authorization": f"Bearer {os.environ['CS_API_KEY']}",
        "Idempotency-Key": "order-78421-marketing",
    },
    json={
        "data_principal": {"email": "a@example.in"},
        "purpose_code": "marketing_email",
        "outcome": "granted",
        "origin": "https://shop.example.in/checkout",
    },
)
artefact_id = r.json()["artefact_id"]`,
  },
]} />

## 4. Verify & archive

Use the artefact id to re-verify consent on every subsequent transaction,
and to produce DPB-facing audit exports.

<CodeTabs tabs={[
  {
    label: 'cURL',
    language: 'bash',
    code: `# Before sending marketing — re-verify the artefact is still active.
curl "https://api.consentshield.in/v1/consent/verify?artefact_id=01HRG5K9X7N5B2VA8P3M9Q4W1T" \\
  -H "Authorization: Bearer $CS_API_KEY"

# Expect: {"status": "active", "expires_at": "2027-04-24T…", ...}`,
  },
]} />

<Callout tone="tip" title="You're done">
That artefact is now the system of record. It survives banner changes,
session drops, and employee churn. To invalidate it, call
`POST /v1/consent/artefacts/{id}/revoke` — **never delete rows directly**.
The buffer tables that back the API are ephemeral; the canonical record
lives in the customer's own R2/S3 bucket, not ConsentShield's database.
</Callout>
